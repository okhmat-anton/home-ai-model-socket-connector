================================================================================
       ТЕХНИЧЕСКОЕ ЗАДАНИЕ: HOME AI MODEL SOCKET CONNECTOR
================================================================================

Назначение:
  Двунаправленный мост между локальной ИИ-моделью и внешним миром.
  Два режима работы:
    • ВХОДЯЩИЙ — пользователь отправляет запрос на сервер, сервер передаёт
      его модели (подключённой по Socket.IO) и возвращает ответ.
    • ИСХОДЯЩИЙ — модель использует сервер как стандартный HTTPS-прокси
      (HTTP CONNECT). Сервер выступает точкой выхода: модель получает
      другой IP (IP сервера) для обращения к внешним ресурсам.
      Аутентификация на прокси — логин и пароль (Proxy-Authorization).
  Модель аутентифицируется на сервере (Socket.IO) по API-ключу.

================================================================================
1. СТЕК ТЕХНОЛОГИЙ
================================================================================
  Язык:           Python 3.11+
  Фреймворк:      FastAPI (async, uvicorn)
  WebSocket:       python-socketio (async-режим)
  HTTP-клиент:     httpx (async)
  Управление:      Makefile + systemd
  Тесты:           pytest + pytest-asyncio + httpx (AsyncClient)
  Конфигурация:    .env файл (python-dotenv), переопределение через ENV
  Контейнеризация: опционально Docker / docker-compose

================================================================================
2. АРХИТЕКТУРА
================================================================================

  РЕЖИМ 1 — ВХОДЯЩИЙ (user → server → model):

  ┌──────────────┐   HTTP / WS    ┌────────────────────────┐  Socket.IO  ┌────────────────┐
  │  Внешний     │ ──────────────► │  FastAPI-сервер        │ ──────────► │  Локальная ИИ  │
  │  пользователь│ ◄────────────── │  :10666                │ ◄────────── │  модель        │
  └──────────────┘                 │  (REST API + Socket.IO)│             └────────────────┘
                                   └────────────────────────┘

    Пользователь отправляет запрос → сервер проксирует его модели
    через Socket.IO → модель отвечает → сервер возвращает ответ.

  РЕЖИМ 2 — ИСХОДЯЩИЙ (model → proxy → внешний мир):

  ┌────────────────┐  HTTP CONNECT  ┌────────────────────────┐  HTTPS   ┌────────────────┐
  │  Локальная ИИ  │ ──────────────►│  HTTPS-прокси          │ ────────►│  Внешний       │
  │  модель        │ ◄──────────────│  :10667                │ ◄────────│  ресурс        │
  └────────────────┘  (proxy auth)  │  (логин + пароль)      │          │  (API, сайты)  │
                                    └────────────────────────┘          └────────────────┘

    Модель настраивает переменные окружения:
      HTTPS_PROXY=http://proxy_user:proxy_pass@server_ip:10667
      HTTP_PROXY=http://proxy_user:proxy_pass@server_ip:10667
    Все HTTP(S)-запросы модели идут через прокси → IP запроса = IP сервера.
    Стандартный протокол HTTP CONNECT для HTTPS-туннелирования.

  Компоненты:
    • FastAPI-приложение — REST-эндпоинты + Socket.IO-сервер (порт 10666)
    • Socket.IO namespace /model — канал связи с моделью (входящий режим)
    • HTTPS-прокси — отдельный asyncio-сервер (порт 10667), Basic Auth
    • systemd unit — автозапуск, перезапуск при падении, watchdog

3. КОНФИГУРАЦИЯ (.env)
================================================================================

  HOST=0.0.0.0                          # адрес прослушивания
  PORT=10666                             # порт сервера
  BASE_MODEL=llama3                      # модель по умолчанию
  MODEL_API_KEY=<secure_random_64>       # API-ключ для аутентификации модели (Socket.IO)
  USER_API_KEY=<secure_random_64>        # API-ключ для аутентификации пользователей (REST)
  SECRET_KEY=<secure_random_64>          # ключ для подписи токенов
  REQUEST_TIMEOUT=1800                   # таймаут запроса к модели (сек, 30 мин)
  MAX_CONCURRENT_REQUESTS=10             # лимит одновременных входящих запросов
  PROXY_PORT=10667                       # порт HTTPS-прокси
  PROXY_USER=proxy                       # логин для прокси
  PROXY_PASSWORD=<secure_random>         # пароль для прокси
  PROXY_ALLOWED_DOMAINS=*                # разрешённые домены для прокси (* = все)
  PROXY_MAX_CONNECTIONS=50               # макс. одновременных соединений через прокси
  LOG_LEVEL=INFO                         # уровень логирования

4. REST API — ЭНДПОИНТЫ
================================================================================

4.1  GET /instruction
     Инструкция по использованию сервиса для ИИ-агента.
     Авторизация: не требуется.
     Формат ответа: text/plain (читаемый текст).
     Ответ 200:
       Возвращает текстовую инструкцию, прочитав которую ИИ-агент
       понимает, как использовать сервис в своём коде. Содержит:
         — Описание двух режимов работы (входящий и исходящий)
         — Примеры кода для отправки запроса через REST API (/ask)
         — Примеры кода для настройки HTTPS-прокси
         — Формат запросов и ответов
         — Описание параметров и кодов ошибок
         — Пример потокового (stream) запроса

     Пример содержимого (сокращённо):
       ---
       # Как использовать сервис Home AI Model Socket Connector

       ## 1. Отправить запрос к ИИ-модели

       ```python
       import httpx

       resp = httpx.post(
           "http://<server_ip>:10666/ask",
           headers={"Authorization": "Bearer <USER_API_KEY>"},
           json={"prompt": "Привет, как дела?", "stream": False}
       )
       print(resp.json()["response"])
       ```

       ## 2. Использовать сервер как HTTPS-прокси (выход в интернет)

       ```python
       import httpx

       client = httpx.Client(
           proxy="http://proxy_user:proxy_pass@<server_ip>:10667"
       )
       resp = client.get("https://api.example.com/data")
       ```

       ## 3. Потоковый запрос (SSE)

       ```python
       import httpx

       with httpx.stream(
           "POST", "http://<server_ip>:10666/ask",
           headers={"Authorization": "Bearer <USER_API_KEY>"},
           json={"prompt": "Расскажи историю", "stream": True}
       ) as resp:
           for line in resp.iter_lines():
               if line.startswith("data: "):
                   print(line[6:])
       ```
       ---

     Текст инструкции хранится в файле src/instruction.txt
     и подставляет актуальные значения (хост, порт, доступные модели)
     динамически при каждом запросе.

4.2  GET /health
     Проверка работоспособности сервера.
     Авторизация: не требуется.
     Ответ 200:
       {
         "status": "ok",
         "version": "1.0.0",
         "uptime_seconds": 3600,
         "connected_models": ["llama3", "mistral"],
         "base_model": "llama3"
       }

4.3  POST /ask
     Отправка запроса к модели.
     Авторизация: Bearer-токен или Basic Auth.
     Тело запроса (JSON):
       {
         "prompt": "Расскажи про квантовые компьютеры",
         "model": "llama3",          // опционально, по умолчанию BASE_MODEL
         "stream": false,            // опционально, потоковый ответ
         "parameters": {             // опционально
           "temperature": 0.7,
           "max_tokens": 2048,
           "top_p": 0.9
         }
       }
     Ответ 200 (stream=false):
       {
         "response": "Квантовые компьютеры — это...",
         "model": "llama3",
         "usage": {
           "prompt_tokens": 12,
           "completion_tokens": 345,
           "total_tokens": 357
         },
         "elapsed_seconds": 4.2
       }
     Ответ 200 (stream=true):
       Server-Sent Events (text/event-stream), каждый chunk:
       data: {"token": "Кванто", "done": false}
       ...
       data: {"token": "", "done": true, "usage": {...}, "elapsed_seconds": 4.2}

     Коды ошибок:
       401 — отсутствует или неверная авторизация
       404 — запрошенная модель не подключена
       408 — таймаут ответа модели (REQUEST_TIMEOUT)
       422 — невалидное тело запроса
       429 — превышен лимит одновременных запросов
       502 — модель недоступна (бэкенд не отвечает)
       500 — внутренняя ошибка сервера

4.4  GET /models
     Список доступных моделей.
     Авторизация: Bearer-токен или Basic Auth.
     Ответ 200:
       {
         "models": [
           {"name": "llama3", "status": "connected", "is_base": true},
           {"name": "mistral", "status": "connected", "is_base": false}
         ]
       }

4.5  POST /auth/token
     Получение Bearer-токена по логину/паролю.
     Тело запроса:
       { "username": "admin", "password": "..." }
     Ответ 200:
       { "access_token": "eyJ...", "token_type": "bearer", "expires_in": 86400 }

================================================================================
5. SOCKET.IO — ВЗАИМОДЕЙСТВИЕ С МОДЕЛЬЮ
================================================================================

  Namespace:   /model
  Транспорт:   WebSocket (fallback: long-polling)
  Порт:        10666 (совместно с HTTP)

  5.1  Подключение модели к серверу
       Событие: connect
       Модель передаёт MODEL_API_KEY в handshake-данных:
         { "api_key": "<MODEL_API_KEY>", "model_name": "llama3" }
       Сервер проверяет ключ и регистрирует модель по имени.
       При невалидном ключе — отклонение подключения.

  5.2  События (сервер → модель) — ВХОДЯЩИЙ РЕЖИМ:
       • "inference_request"   — запрос на генерацию от пользователя
         Payload: { request_id, prompt, parameters }
       • "ping"                — keepalive
         Payload: { timestamp }

  5.3  События (модель → сервер) — ответы на входящие запросы:
       • "inference_response"  — результат генерации
         Payload: { request_id, response, usage, elapsed }
       • "inference_chunk"     — chunk при потоковой генерации
         Payload: { request_id, token, done }
       • "inference_error"     — ошибка генерации
         Payload: { request_id, error, code }
       • "pong"                — ответ на keepalive
         Payload: { timestamp }

  5.4  Реконнект
       При разрыве соединения модель автоматически переподключается
       с экспоненциальной задержкой (1 с, 2 с, 4 с, ..., макс. 60 с).
       Сервер хранит очередь pending-запросов и переотправляет их после
       реконнекта, если не истёк таймаут.

6. АУТЕНТИФИКАЦИЯ И БЕЗОПАСНОСТЬ
================================================================================

  6.1  Аутентификация модели (Socket.IO)
       • Модель передаёт MODEL_API_KEY при подключении в handshake.
       • Сервер сверяет ключ с MODEL_API_KEY из .env.
       • При невалидном ключе — соединение отклоняется.
       • Один ключ = одна модель. Для нескольких моделей — список ключей
         в формате MODEL_API_KEYS=key1:model1,key2:model2 (опционально).

  6.2  Аутентификация пользователя (REST API)
       • HTTP-эндпоинты (/ask, /models) защищены заголовком:
           Authorization: Bearer <USER_API_KEY>
       • USER_API_KEY задаётся в .env.
       • Опционально: POST /auth/token для получения JWT по логину/паролю
         (если нужна многопользовательская схема).

  6.3  Аутентификация и безопасность HTTPS-прокси (порт 10667)
       • Аутентификация: Proxy-Authorization Basic (логин + пароль).
         Модель настраивает прокси стандартным способом:
           HTTPS_PROXY=http://proxy_user:proxy_pass@server_ip:10667
       • При неверных credentials — 407 Proxy Authentication Required.
       • PROXY_ALLOWED_DOMAINS — белый список доменов, к которым можно
         обращаться через прокси (* = все, по умолчанию).
       • Запрещены запросы к localhost / 127.0.0.1 / приватным сетям
         (защита от SSRF) — настраивается.
       • PROXY_MAX_CONNECTIONS — макс. одновременных соединений.
       • Лимит на количество запросов в минуту через прокси.

  6.4  Общая безопасность
       • Rate limiting: настраиваемый лимит запросов в минуту на клиента.
       • CORS: настраиваемый список разрешённых origins.
       • Ключи хранятся в .env, файл исключён из git через .gitignore.
       • SECRET_KEY используется для подписи JWT-токенов (HS256).

                          sudo systemctl restart home-ai-connector

                          1. git pull --ff-only origin main
                          2. venv/bin/pip install -r requirements.txt
                          3. make test
                          4. sudo systemctl restart home-ai-connector
                        При провале тестов — откат (git checkout @{1}),
                        перезапуск прежней версии, вывод ошибки.

  make test           — Запуск автотестов:
                          venv/bin/pytest tests/ -v --tb=short

  make logs           — Просмотр логов сервера в реальном времени:
                          journalctl -u home-ai-connector -f

  make status         — Статус systemd-сервиса:
                          systemctl status home-ai-connector

  make gen-keys       — Генерация новых API-ключей (MODEL_API_KEY, USER_API_KEY)
                        и запись их в .env:
                          python3 -c "import secrets; print(secrets.token_hex(32))"

  make uninstall      — Удаление сервиса:
                          1. sudo systemctl stop home-ai-connector
                          2. sudo systemctl disable home-ai-connector
                          3. sudo rm /etc/systemd/system/home-ai-connector.service
                          4. sudo systemctl daemon-reload

8. ЛОГИКА ОБРАБОТКИ ЗАПРОСОВ
================================================================================

  ВХОДЯЩИЙ РЕЖИМ (user → server → model):

  1. Клиент отправляет POST /ask с prompt и (опционально) именем модели.
  2. Сервер проверяет авторизацию (Bearer USER_API_KEY).
  3. Если модель не указана — используется BASE_MODEL из конфигурации.
  4. Сервер проверяет, что запрошенная модель подключена через Socket.IO.
     Если нет — возвращает 404 с описанием доступных моделей.
  5. Проверяется лимит одновременных запросов (семафор asyncio.Semaphore).
     При превышении — 429.
  6. Сервер генерирует уникальный request_id (UUID4) и отправляет
     событие "inference_request" модели через Socket.IO.
  7. Сервер ожидает ответ (asyncio.Event / asyncio.Queue).
     — При stream=false: ждёт полный "inference_response".
     — При stream=true: проксирует "inference_chunk" как SSE.
  8. Если ответ не получен за REQUEST_TIMEOUT (по умолчанию 30 минут) —
     возвращается 408 Timeout.
  9. При получении "inference_error" — возвращается соответствующий HTTP-код.
  10. Результат возвращается клиенту с метаданными (usage, elapsed).

  ИСХОДЯЩИЙ РЕЖИМ (model → proxy → внешний мир):

  Модель общается с прокси по стандартному HTTP-протоколу (порт 10667).
  Socket.IO для этого не используется.

  1. Модель отправляет HTTP-запрос через прокси (CONNECT для HTTPS,
     прямой запрос для HTTP).
  2. Прокси проверяет Proxy-Authorization (Basic: login + password).
     При неверных credentials — 407 Proxy Authentication Required.
  3. Прокси проверяет целевой домен по PROXY_ALLOWED_DOMAINS.
     Приватные сети (localhost, 10.*, 192.168.*, 172.16.*) заблокированы.
     При запрете — 403 Forbidden.
  4. Прокси устанавливает соединение с целевым сервером:
     — Для HTTPS: TCP-туннель через CONNECT (прокси не видит содержимое).
     — Для HTTP: прокси выполняет запрос и проксирует ответ.
     — IP-адрес = IP сервера (модель скрыта за прокси).
  5. Данные передаются между моделью и целевым сервером.
  6. Все запросы логируются (target host, метод, статус, байты, elapsed).

9. АВТОТЕСТЫ
================================================================================

  Фреймворк: pytest + pytest-asyncio
  Расположение: tests/

  Покрытие:
    • test_instruction.py
        — GET /instruction возвращает 200 и text/plain
        — Ответ содержит примеры кода (python, httpx)
        — Ответ содержит актуальный порт и список моделей

    • test_health.py
        — GET /health возвращает 200 и корректную структуру
        — Проверка полей status, version, uptime_seconds

    • test_auth.py
        — POST /auth/token с валидными credentials → 200 + токен
        — POST /auth/token с невалидными credentials → 401
        — Запрос к /ask без токена → 401
        — Запрос к /ask с истёкшим токеном → 401

    • test_ask.py
        — POST /ask с валидным prompt → 200 + response
        — POST /ask с указанием конкретной модели → корректный роутинг
        — POST /ask без модели → использование BASE_MODEL
        — POST /ask с несуществующей моделью → 404
        — POST /ask с невалидным телом (пустой prompt) → 422
        — POST /ask при отсутствии подключённых моделей → 502
        — POST /ask stream=true → корректный SSE-поток

    • test_models.py
        — GET /models возвращает список подключённых моделей
        — Проверка флага is_base

    • test_socketio.py
        — Подключение модели с валидным API-ключом → регистрация
        — Подключение с невалидным ключом → отклонение
        — Отключение модели → удаление из списка
        — Отправка inference_request → получение inference_response
        — Таймаут запроса → 408
        — Реконнект модели → pending-запросы переотправляются

    • test_proxy.py
        — CONNECT к разрешённому домену с валидным auth → 200 туннель
        — CONNECT без Proxy-Authorization → 407
        — CONNECT с неверными credentials → 407
        — CONNECT к заблокированному домену → 403
        — CONNECT к localhost / приватной сети → 403 (SSRF)
        — Превышение PROXY_MAX_CONNECTIONS → 503
        — HTTP-запрос через прокси → корректное проксирование
        — Лимит запросов в минуту → 429

    • test_concurrency.py
        — Параллельные запросы в пределах лимита → все обработаны
        — Превышение MAX_CONCURRENT_REQUESTS → 429

    • test_config.py
        — Загрузка конфигурации из .env
        — Переопределение через переменные окружения

  Запуск:
    make test           — полный прогон
    make update         — прогон после обновления (откат при провале)

10. ЛОГИРОВАНИЕ И МОНИТОРИНГ
================================================================================

  • Формат логов: JSON (structlog) для удобного парсинга.
  • Уровни: DEBUG, INFO, WARNING, ERROR (настраивается через LOG_LEVEL).
  • Логируемые события:
      — Запуск / остановка сервера
      — Подключение / отключение модели
      — Входящий запрос (метод, путь, клиент, request_id)
      — Результат запроса (статус, elapsed, tokens)
      — Ошибки аутентификации
      — Таймауты
      — Ошибки модели
      — Прокси: подключение, target host, байты, elapsed
      — Прокси: заблокированные запросы (SSRF, домен вне белого списка)
      — Прокси: ошибки аутентификации
  • Вывод: stdout (перехватывается journalctl через systemd).
  • Просмотр: make logs
